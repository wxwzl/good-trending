# ADMEMS 架构设计方法论

基于《一线架构师实践指南》(温昱)的系统化软件架构设计方法，帮助架构师从需求出发，逐步设计出可落地的系统架构。

## 核心理念

### 架构师的6个经典困惑

1. **架构设计的"驱动力"是什么？** - 需求如何驱动架构设计
2. **架构设计做到什么程度？** - 细化到什么层级才算完成
3. **概念架构怎么设计？** - 大系统如何从全局视角进行设计
4. **多视图方法如何落地？** - 5视图方法具体怎么用
5. **非功能需求如何处理？** - 质量属性如何影响架构决策
6. **架构如何验证？** - 设计完成后如何确保正确性

### 4个核心主张

1. **方法体系是大趋势** - 经验主义不可持续，需要系统化方法
2. **质疑驱动的架构设计** - 通过提问引导设计思考
3. **多阶段还是多视图？** - 阶段是时间维度，视图是空间维度，两者结合
4. **内置最佳实践** - 方法中融入业界最佳实践

## 阶段一：Pre-Architecture（需求结构化阶段）

### 目的

在正式设计架构之前，全面理解和结构化需求，识别关键需求作为架构设计的驱动力。

### 核心原则

> **关键需求决定架构，其余需求验证架构**

### 4个步骤

#### 步骤1：需求结构化（ADMEMS矩阵）

将需求按两个维度进行分类：

| | 功能需求 | 质量需求 | 约束需求 |
|---|---|---|---|
| **业务级需求** | 业务目标 | 业务质量 | 业务约束 |
| **用户级需求** | 用户功能 | 用户体验 | 用户约束 |
| **开发级需求** | 系统功能 | 系统质量 | 技术约束 |

**ADMEMS矩阵示例：**

```
┌──────────────┬─────────────────┬─────────────────┬─────────────────┐
│              │    功能需求      │    质量需求      │    约束需求      │
├──────────────┼─────────────────┼─────────────────┼─────────────────┤
│ 业务级需求    │ 核心业务流程     │ 响应时间 < 2s    │ 合规性要求       │
│              │ 关键业务场景     │ 系统可用性 99.9% │ 预算限制         │
├──────────────┼─────────────────┼─────────────────┼─────────────────┤
│ 用户级需求    │ 用户操作流程     │ 界面响应速度     │ 用户技术水平     │
│              │ 用户交互场景     │ 操作便捷性       │ 设备限制         │
├──────────────┼─────────────────┼─────────────────┼─────────────────┤
│ 开发级需求    │ 系统功能模块     │ 可维护性         │ 技术栈选型       │
│              │ 接口设计        │ 可扩展性         │ 团队技能         │
└──────────────┴─────────────────┴─────────────────┴─────────────────┘
```

#### 步骤2：分析约束影响

**约束分类理论：**

| 约束类型 | 描述 | 架构影响 |
|---|---|---|
| **直接约束** | 直接限制技术选型或架构形式 | 必须遵守，不可绕过 |
| **间接约束** | 通过影响其他需求间接影响架构 | 需要分析和推导 |
| **隐含约束** | 未明确说明但实际存在 | 需要主动识别 |

**约束分析法则：**

1. **推导法则** - 从显式约束推导隐式约束
2. **查漏法则** - 检查是否遗漏重要约束

**约束影响分析示例：**

```
输入：约束列表
输出：约束对架构的具体影响

约束: 必须使用现有Oracle数据库
├── 直接影响: 数据层必须支持Oracle
├── 间接影响: ORM框架选型受限
└── 隐含约束: 需要DBA参与数据架构设计

约束: 系统需要7×24小时运行
├── 直接影响: 需要高可用架构
├── 间接影响: 需要考虑灰度发布
└── 隐含约束: 运维监控体系要求
```

#### 步骤3：确定关键质量目标

**5大原则：**

1. **分类合适+必要扩充** - 质量属性分类要完整
2. **考虑多方涉众** - 不同角色关注不同质量
3. **检查性思维** - 质量目标要可验证
4. **识别矛盾+划定优先级** - 质量属性之间可能冲突
5. **严格程度符合领域与规模特点** - 不同系统要求不同

**常见质量属性分类：**

```
质量属性
├── 运行期质量
│   ├── 性能 (响应时间、吞吐量)
│   ├── 可用性 (MTBF、MTTR)
│   ├── 安全性 (认证、授权、加密)
│   ├── 可伸缩性 (水平扩展、垂直扩展)
│   └── 易用性 (学习曲线、操作效率)
└── 开发期质量
    ├── 可维护性 (定位问题、修改成本)
    ├── 可扩展性 (添加功能、集成)
    ├── 可重用性 (组件复用)
    ├── 可测试性 (单元测试、集成测试)
    └── 可移植性 (跨平台、跨环境)
```

**质量目标示例：**

| 质量属性 | 目标值 | 优先级 | 验证方法 |
|---|---|---|---|
| 响应时间 | 核心接口 < 200ms | 高 | 性能测试 |
| 并发用户 | 支持10000在线用户 | 高 | 压力测试 |
| 可用性 | 99.9% | 高 | 监控统计 |
| 数据安全 | 敏感数据加密存储 | 高 | 安全审计 |

#### 步骤4：确定关键功能

**4条规则：**

1. **核心功能优先** - 识别系统的核心业务价值
2. **高风险功能关注** - 技术难度大或风险高的功能
3. **高频使用功能** - 用户最常用的功能
4. **关键路径覆盖** - 覆盖完整业务流程

**关键功能识别矩阵：**

| 功能 | 业务价值 | 技术风险 | 使用频率 | 优先级 |
|---|---|---|---|---|
| 用户认证 | 高 | 中 | 高 | ★★★ |
| 订单处理 | 高 | 高 | 高 | ★★★ |
| 支付集成 | 高 | 高 | 中 | ★★★ |
| 报表统计 | 中 | 低 | 低 | ★ |

## 阶段二：Conceptual Architecture（概念架构阶段）

### 目的

基于关键需求，从全局视角设计系统的高层架构，明确系统的大方向。

### 核心原则

> **重大需求塑造概念架构**

### 3个步骤

#### 步骤1：初步设计（鲁棒图）

**鲁棒图三元素：**

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   边界对象   │ ──→  │   控制对象   │ ──→  │   实体对象   │
│  (Boundary)  │ ←──  │  (Control)  │ ←──  │   (Entity)  │
└─────────────┘      └─────────────┘      └─────────────┘
     UI/接口            业务逻辑            数据/领域
```

**鲁棒图建模规则：**

- 边界对象 ↔ 控制对象 ↔ 实体对象
- 边界对象 ✗ 实体对象（不能直接通信）
- 同类对象之间可以交互

**鲁棒图示例：**

```
用户登录场景：

[登录页面] ──→ [登录控制器] ──→ [用户实体]
     ↑              │               │
     │              ↓               ↓
[错误提示] ←── [会话管理] ←── [权限实体]
```

**10条鲁棒图经验：**

1. 遵守建模规则
2. 简化建模语法
3. 遵循3种元素的发现思路
4. 增量建模
5. 实体对象 ≠ 持久化对象
6. 只对关键功能画鲁棒图
7. 每个鲁棒图有2~5个控制对象
8. 勿关注细节
9. 勿过分关注UI
10. 鲁棒图 ≠ 用例规约的可视化

#### 步骤2：高层分割

**两种实践套路：**

**套路1：切系统为系统**

适用于：超大型系统、分布式系统

```
大系统
├── 子系统A
│   ├── 模块A1
│   └── 模块A2
├── 子系统B
│   ├── 模块B1
│   └── 模块B2
└── 子系统C
    └── 模块C1
```

**套路2：切系统为子系统**

适用于：中等规模系统

```
系统
├── 子系统/模块A
├── 子系统/模块B
├── 子系统/模块C
└── 子系统/模块D
```

**分层架构设计：**

| 分层方式 | 描述 | 适用场景 |
|---|---|---|
| **Layer（逻辑层）** | 逻辑职责划分 | 业务复杂度高的系统 |
| **Tier（物理层）** | 物理部署划分 | 需要独立扩展的系统 |
| **通用性分层** | 通用性程度划分 | 产品化系统 |

**逻辑层（Layer）示例：**

```
┌─────────────────────────────────────┐
│           表现层 (Presentation)       │  用户交互
├─────────────────────────────────────┤
│           应用层 (Application)        │  用例编排
├─────────────────────────────────────┤
│           领域层 (Domain)            │  业务核心
├─────────────────────────────────────┤
│           基础设施层 (Infrastructure) │  技术支撑
└─────────────────────────────────────┘
```

**物理层（Tier）示例：**

```
┌──────────────┐
│   客户端层    │  Web/App/小程序
└──────┬───────┘
       │
┌──────↓───────┐
│   网关层      │  API Gateway
└──────┬───────┘
       │
┌──────↓───────┐
│   应用层      │  微服务集群
└──────┬───────┘
       │
┌──────↓───────┐
│   数据层      │  数据库/缓存
└──────────────┘
```

**通用性分层示例：**

```
┌─────────────────────────────────────┐
│        业务专用层 (Specific)          │  特定业务
├─────────────────────────────────────┤
│        通用业务层 (Generic)           │  可复用业务
├─────────────────────────────────────┤
│        技术平台层 (Platform)          │  技术框架
├─────────────────────────────────────┤
│        基础设施层 (Infrastructure)     │  底层支撑
└─────────────────────────────────────┘
```

#### 步骤3：考虑非功能需求

在概念架构阶段就要考虑非功能需求的影响：

- 高可用 → 考虑冗余设计
- 高性能 → 考虑缓存策略
- 高并发 → 考虑水平扩展
- 安全性 → 考虑安全分层

### 视图1：逻辑架构

**目的**：描述系统的功能分解和模块结构

**设计要素：**

```
逻辑架构设计
├── 分层 (Layer)
│   └── 职责划分，降低耦合
├── 分区 (Partition)
│   └── 同层功能分组
└── 机制 (Mechanism)
    └── 横切关注点提取
```

**划分子系统的3种策略：**

| 策略 | 描述 | 示例 |
|---|---|---|
| **分层细化** | 每层进一步细分 | 表现层→页面、组件、样式 |
| **分区引入** | 同层按功能分区 | 业务层→订单区、用户区、商品区 |
| **机制提取** | 提取通用机制 | 日志、认证、缓存、异常处理 |

**划分子系统的4个重要原则：**

1. **高内聚** - 模块内部紧密相关
2. **低耦合** - 模块之间松散依赖
3. **单一职责** - 每个模块只做一件事
4. **接口隔离** - 依赖最小接口

**逻辑架构示例：**

```

│  └─────────┘ └─────────┘ └─────────┘ └─────────┘           │
├─────────────────────────────────────────────────────────────┤
│                        领域层                                │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐           │
│  │用户聚合根│ │订单聚合根│ │商品聚合根│ │支付聚合根│           │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘           │

### 视图5：数据架构

**目的**：描述系统的数据组织和存储策略

**数据分布的6种策略：**

| 策略 | 描述 | 适用场景 |
|---|---|---|
| **独立Schema** | 每个服务独立数据库 | 微服务架构 |
| **集中式** | 单一共享数据库 | 简单系统、事务要求高 |
| **分区** | 数据按规则分散存储 | 大数据量、地域分布 |
| **复制** | 数据冗余到多个节点 | 高可用、读多写少 |
| **子集** | 存储数据的部分副本 | 报表、分析 |
| **重组** | 按用途重新组织数据 | 数据仓库 |

**数据分布策略选择原则：**

1. **合适原则** - 根据实际需求选择，不追求完美
2. **综合原则** - 多策略组合使用
3. **优化原则** - 持续优化数据访问性能

**数据架构示例：**

```
